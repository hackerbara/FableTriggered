# Claude Code on Windows: Distribution, Install, Update, and Signing — Research Report

*Compiled 2026-07-04. All claims sourced; confidence noted where evidence is partial.*

## 1. Distribution channels on Windows

As of mid-2026, Anthropic recommends the **native installer** as the primary path on every platform, including Windows. The official docs present it first, labeled "Recommended":

- PowerShell: `irm https://claude.ai/install.ps1 | iex`
- CMD: `curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd`

Alternative channels documented: **WinGet** (`winget install Anthropic.ClaudeCode`, does **not** auto-update), and **npm** (`npm install -g @anthropic-ai/claude-code`). There is no Homebrew equivalent on Windows (Homebrew tab is macOS/Linux only). WSL is supported by running the Linux installer (`curl -fsSL https://claude.ai/install.sh | bash`) inside the WSL distro.

**npm package on Windows ships a native binary, not JS.** Confirmed directly from the npm registry (`registry.npmjs.org/@anthropic-ai/claude-code/latest`, v2.1.201): `optionalDependencies` includes `@anthropic-ai/claude-code-win32-x64` and `@anthropic-ai/claude-code-win32-arm64`, alongside darwin/linux variants. Docs state explicitly: "The npm package installs the same native binary as the standalone installer... The installed `claude` binary does not itself invoke Node." A postinstall step links the platform-specific optional-dependency package into place. If optional deps are disabled (`--omit=optional`), install fails with "Could not find native binary package at runtime" — there is no JS fallback.

Sources: [Advanced setup — code.claude.com/docs/en/setup](https://code.claude.com/docs/en/setup), npm registry JSON fetch, [npmjs.com/package/@anthropic-ai/claude-code-win32-x64](https://www.npmjs.com/package/@anthropic-ai/claude-code-win32-x64?activeTab=versions).

## 2. Install locations and auto-update mechanics

**Locations** (from official docs' uninstall section and corroborated by GitHub issues):
- Launcher/stub binary: `%USERPROFILE%\.local\bin\claude.exe`
- Versioned binaries: `%USERPROFILE%\.local\share\claude\versions\<version>` — e.g. `~/.local/share/claude/versions/2.1.51`, `.../2.1.52`, each a full-size copy (one issue reports a version file at 235,398,304 bytes, ≈224 MiB, consistent with a Bun-compiled binary bundling the whole app + runtime).
- Temp download staging during install: `%USERPROFILE%\.claude\downloads` (per the bootstrap script, see below).

**Bootstrap script mechanics** (fetched directly — `https://claude.ai/install.ps1` 302-redirects to `https://downloads.claude.ai/claude-code-releases/bootstrap.ps1`): it resolves the latest version from `.../claude-code-releases/latest`, fetches `manifest.json` for that version, downloads `.../<version>/<platform>/claude.exe` (platform = `win32-arm64` or `win32-x64` by detected architecture) into the temp downloads folder as `claude-<version>-<platform>.exe`, then runs `& $binaryPath install $Target` — i.e., **the downloaded binary installs itself** (sets up the launcher stub and shell integration), and the temp file is deleted afterward. The install script always re-downloads the current latest installer regardless of target version, to make sure the installer logic itself is current.

**Auto-update**: native installs "automatically update in the background... Claude Code checks for updates on startup and periodically while running. Updates download and install in the background, then take effect the next time you start Claude Code." WinGet and npm installs do **not** auto-update by default (`winget upgrade Anthropic.ClaudeCode` / `npm install -g @anthropic-ai/claude-code@latest` manually). A release-channel setting (`autoUpdatesChannel`: `latest`/`stable`) and a `minimumVersion` floor exist; `DISABLE_AUTOUPDATER`/`DISABLE_UPDATES` env vars can turn it off entirely.

**Real-world update failures are common and instructive for a patcher**, per GitHub issues:
- **#28285** — `claude update` downloads the new version into `versions/` correctly but fails to copy it over `~/.local/bin/claude.exe`; the stub silently keeps pointing at the old version. Reporter explicitly ruled out file-locking (ran update from a separate terminal) — so the "promote versions/X to bin/claude.exe" step itself is fragile, independent of locking.
- **#51954** — a Windows-specific service (`CoworkVMService`, tied to the Desktop/Cowork MSIX packaging) can hold an **exclusive lock** on the exe during update, leaving the machine needing a service restart or reboot before `claude.exe` becomes runnable again (Windows won't execute a locked/mid-swap binary; users saw an "Open with..." dialog).
- **#69884** — during some auto-update races, the launcher stub at `.local\bin\claude.exe` ends up being a **bare, un-repackaged Bun v1.4.0 runtime** instead of Claude Code (launches Bun's own help text). This confirms the updater's "download → self-install via `install` subcommand → delete temp file" pipeline can leave the target directory in an inconsistent state under Windows file-handle contention.
- **#21365** — separately, the installer historically did not add `%USERPROFILE%\.local\bin` to `PATH` automatically, producing `CommandNotFoundException` right after a "successful" install (a warning is now printed; manual `PATH` edit needed).

Confidence: high for paths/mechanism (multiple independent sources agree); the exact atomic-replace algorithm (rename-then-delete vs. direct overwrite) is **not documented anywhere** — only inferred from failure reports. Treat as unverified implementation detail.

Sources: [Advanced setup](https://code.claude.com/docs/en/setup), bootstrap.ps1 (fetched live), [#28285](https://github.com/anthropics/claude-code/issues/28285), [#51954](https://github.com/anthropics/claude-code/issues/51954), [#69884](https://github.com/anthropics/claude-code/issues/69884), [#21365](https://github.com/anthropics/claude-code/issues/21365).

## 3. Binary format: confirmed Bun-compiled

**Confirmed, not just inferred.** GitHub issue **#69884** ("Windows: .local\bin\claude.exe installs as plain Bun v1.4.0 instead of Claude Code") states directly: Anthropic builds the Windows exe via `bun build --compile`, and that this command "always sets `InternalName: bun` — there is no build flag to customize this" in the PE version resource. This is exactly analogous to your macOS finding (Bun-compiled Mach-O) — same toolchain, same compile-time metadata quirk, just surfaced through Windows' PE `InternalName` field instead of Mach-O load commands. The reported workaround in that issue is `rcedit claude.exe --set-version-string "InternalName" "claude"`, confirming the exe is a standard PE that tools like `rcedit` (an open-source PE resource editor used commonly for Electron/Bun apps) can manipulate.

A third-party project, `winfunc/claude-code-single-binary`, exists specifically to build Claude Code into single-file Windows/Mac/Linux executables with bundled vendor deps — further circumstantial confirmation that the upstream build is Bun's `--compile` pipeline (Bun's own docs describe `bun build --compile --target=bun-windows-x64` producing ~100 MB+ single-file PE executables with an embedded runtime, matching the ~224 MB versioned binary size observed in #28285's directory listing).

Confidence: **high** that it's Bun-compiled (direct maintainer/community confirmation + metadata artifact), but the exact internal layout of the embedded JS bundle in the PE (where in the file it's appended, whether it's an appended blob after the last PE section like on Mach-O, checksums, etc.) is **not documented publicly** — you'd need to reverse-engineer a real `claude.exe` the same way you presumably reverse-engineered the Mach-O, or look at Bun's own compiler source (`bun build --compile` implementation in oven-sh/bun) for the PE embedding format.

Sources: [#69884](https://github.com/anthropics/claude-code/issues/69884), [winfunc/claude-code-single-binary](https://github.com/winfunc/claude-code-single-binary), [Bun: Building Windows](https://bun.com/docs/project/building-windows), [Bun: Single-file executable](https://bun.com/docs/bundler/executables).

## 4. Signing

**Confirmed Authenticode-signed.** Official docs, "Binary integrity and code signing" section: *"Windows: signed by 'Anthropic, PBC'. Verify with `Get-AuthenticodeSignature .\claude.exe`."* (macOS equivalent: signed "Anthropic PBC" + Apple-notarized, verified via `codesign`; Linux binaries are unsigned individually but covered by a signed manifest + package-repo GPG signing.) Additionally, each release publishes a `manifest.json` with SHA256 checksums, itself signed with a detached GPG signature (Anthropic key fingerprint `31DD DE24 DDFA B679 F42D 7BD2 BAA9 29FF 1A7E CACE`) — so there are **two independent integrity layers** on Windows: the Authenticode signature on the exe itself, and the GPG-signed manifest checksum.

**Real-world signing has broken in production**: GitHub issue **#10608** documents that Windows builds after v2.0.24 (versions 2.0.25–2.0.29) shipped signed with an **expired certificate**, causing `ResourceUnavailable` failures under Group Policy signature-whitelisting enforcement (`This program is blocked by group policy`). This confirms Anthropic's Windows signing pipeline is a real, actively-maintained Authenticode chain that can and has failed operationally — not a one-time triviality.

**Practical effect of modifying a signed exe**: Authenticode signatures are detached/appended metadata (an embedded PKCS#7 blob in a PE certificate-table entry) — any byte modification elsewhere in the file **invalidates the signature but does not itself block execution**. Unlike macOS Gatekeeper (which can hard-block execution of a tampered/ad-hoc-resigned binary depending on quarantine attributes and policy), Windows SmartScreen's reputation model is more forgiving of tampering per se but punishes **absence of a valid, reputable signature**: an unsigned or invalid-signature exe is far more likely to trigger a SmartScreen "Windows protected your PC" interstitial than a validly-signed one, especially for a freshly-modified/low-reputation file hash. Practically: **patching `claude.exe` will break its Authenticode signature** (there's no ad-hoc equivalent on Windows the way `codesign --sign -` works on macOS), and re-signing requires a real code-signing certificate (EV or OV) that only Anthropic possesses — you cannot resign as "Anthropic, PBC." Options are (a) ship unsigned and accept SmartScreen friction / possibly Defender heuristic flags (Bun binaries specifically have a documented history of Windows Defender false-positive "trojan" flags even when officially built — see `oven-sh/bun#16981`), or (b) self-sign with your own cert (users would see "the publisher could not be verified" rather than "Anthropic, PBC," and would need to trust a new publisher).

**Directly relevant technical risk found**: `oven-sh/bun#20109`, "Bun Single-File Executables Breaking when signed on Windows" — reports that a `bun build --compile` executable **runs fine unsigned but breaks after Authenticode `signtool` signing**, and that Bun 1.3 added "Authenticode signature stripping" as a mitigation for post-build signing workflows. This strongly suggests Bun's Windows PE embedding (likely an appended data blob carrying the bundled JS/assets, similar to the Mach-O `__BUN` custom load-command segment on your macOS tool's target) can collide with where Authenticode inserts/reads its certificate table, and that Bun's own tooling now has to actively strip/rewrite something before (re-)signing works. **This is the single most important unresolved technical fact for your Windows port**: if signing a bun-compiled PE after modification requires stripping something Bun itself only recently automated, a naive byte-patch-then-`signtool sign` pipeline could produce a broken binary, not just an unsigned one. I could not find the maintainers' technical root-cause writeup (issue thread had no comments in the fetched view) — you should reproduce this locally with a throwaway self-signed cert before committing to a design. **Confidence: medium** — the symptom is corroborated by a real Bun issue, but the underlying mechanism (why signing breaks it) is not confirmed by Bun maintainers in what I could retrieve.

Sources: [Advanced setup — Binary integrity and code signing](https://code.claude.com/docs/en/setup), [#10608](https://github.com/anthropics/claude-code/issues/10608), [oven-sh/bun#20109](https://github.com/oven-sh/bun/issues/20109), [oven-sh/bun#16981 (Defender false positive)](https://github.com/oven-sh/bun/issues/16981).

## 5. Requirements / Windows-specific quirks

- **Shell**: Bash, Zsh, PowerShell, or CMD are all listed as supported shells. On native Windows specifically: **Git for Windows is optional, not required.** If present, Claude Code uses Git Bash for its Bash tool; if absent, it falls back to a native **PowerShell tool** (`CLAUDE_CODE_USE_POWERSHELL_TOOL` env var toggles this, rolling out progressively even when Git Bash is present). WSL setups don't need Git for Windows at all (they use the Linux installer/toolchain directly).
- **Sandboxing not supported on native Windows.** The official setup comparison table explicitly marks native Windows sandboxing as "Not supported," vs. "Supported" for WSL2. Confirmed further by issue **#46740** ("Native sandbox support for Windows (non-WSL)"), closed as not planned: Claude Code's sandbox uses Seatbelt (macOS) / bubblewrap (Linux, incl. WSL2) with no Windows OS-level equivalent; without it, permission "deny" rules only constrain Claude's built-in tools, not raw Bash/PowerShell commands.
- **OS floor**: Windows 10 1809+ / Windows Server 2019+, x64 or ARM64, 4GB+ RAM.
- **File locking of the running exe** is a real, documented pain point on Windows (unlike POSIX where you can unlink/replace a running binary freely): issues #28285, #51954, #69884 above all trace back to Windows' mandatory file locking on in-use executables interacting badly with the updater's replace step, and WinGet's own upgrade note in the docs says outright: *"On WinGet the upgrade may fail while Claude Code is running because Windows locks the executable. In that case Claude Code shows the manual command instead."* — i.e., Anthropic itself works around this by detecting the lock and falling back to a manual instruction rather than solving in-place replacement.
- **ripgrep**: bundled ("usually included"); Alpine/musl needs manual `libgcc`/`libstdc++`/`ripgrep` — not Windows-relevant, but shows the binary embeds a native ripgrep dependency alongside the Bun runtime, which matters if you're inventorying everything bundled inside the compiled exe.

Sources: [Advanced setup](https://code.claude.com/docs/en/setup), [#46740](https://github.com/anthropics/claude-code/issues/46740).

## 6. Native vs. WSL usage split

I could **not find hard usage-share data** (no Anthropic telemetry blog post, no representative poll). What's available is directional and qualitative only:
- Anthropic's own docs now present native Windows as co-equal/first-listed alongside WSL, with a decision table framed around feature needs (sandboxing → WSL2; Windows-native tooling → native) rather than steering everyone to WSL.
- Secondary-source guides (e.g., ClaudeLab, various 2026 "how to install" blogs) characterize native install as "the better choice for the vast majority of users... faster, simpler, and perfectly sufficient for everyday work," recommending WSL mainly for people who already live in WSL, need Docker, or need sandboxing.
- This is a shift from Claude Code's 2024–2025 early days, when WSL was the only realistic path on Windows (before a native Windows build existed at all).

**Confidence: low-medium** on the actual split — this is aggregated blog/community sentiment, not measured data. Treat "native is now dominant/recommended for typical users, WSL persists for power users needing sandboxing/Docker/Linux toolchains" as a reasonable working assumption, not a hard number.

Sources: [Advanced setup](https://code.claude.com/docs/en/setup) (decision table), [Claude Lab — Native vs WSL2 guide](https://claudelab.net/en/articles/claude-code/claude-code-windows-native-wsl2-complete-guide) (secondary/unverified).

## Implications for a binary-patching tool on Windows

1. **Target format transfers directly, but re-verify the embedding.** It's the same Bun `bun build --compile` pipeline as macOS, so your "parse the Bun module graph, patch exact byte ranges" approach is conceptually portable — but the container is PE, not Mach-O, so the location/structure of the embedded JS bundle (offset, trailer format, any length-prefix or checksum Bun writes) needs independent reverse-engineering. Don't assume identical layout just because it's the "same compiler."

2. **Signing is your biggest open risk, not a solved problem.** You cannot ad-hoc-sign on Windows the way `codesign -s -` works on macOS — Windows has no local, self-issued-equivalent trust anchor that avoids all friction. Realistic options: ship unsigned (accept SmartScreen/Defender friction, worse for a tool already modifying a security-sensitive CLI) or self-sign with your own cert (still shows "unknown publisher," and per `oven-sh/bun#20109`, naively re-signing a Bun-compiled exe **may break it outright**, independent of trust UX). Prototype sign-after-patch on a real Windows box before designing around it — this needs hands-on validation, the public record doesn't explain the mechanism.

3. **PE metadata is separately hackable and separately fragile.** Bun hardcodes `InternalName: bun` in the version resource regardless of target platform; Anthropic apparently doesn't override it either (per #69884, a real shipped stub was observed literally reporting as bare Bun). Tools like `rcedit` can rewrite PE version-resource strings post-compile — useful if you want your patched binary to self-identify distinctly, but confirms the PE resource table is mutable territory separate from the code/bundle payload.

4. **Design your installer/updater to tolerate Windows file locking — don't assume POSIX rename-over-open-file semantics.** macOS/Linux let you unlink or replace a running executable's inode while it's executing; Windows generally won't let you overwrite/delete a file that's mapped/running. Anthropic's own updater has repeatedly shipped bugs here (silent no-op replace, MSIX-service exclusive locks requiring reboot, stub left as raw Bun after a race). Your Windows port needs an explicit rename-old/write-new/atomic-`MoveFileEx`-with-`MOVEFILE_REPLACE_EXISTING`-on-next-boot style pattern (or terminate-and-relaunch), and should detect and gracefully report the locked-file case rather than silently failing like #28285/#51954 show upstream doing.

5. **Know your uninstall/reinstall targets**: stub at `%USERPROFILE%\.local\bin\claude.exe`, versions at `%USERPROFILE%\.local\share\claude\versions\<version>`, config/state at `%USERPROFILE%\.claude` and `%USERPROFILE%\.claude.json` — these are the exact paths your tool will need to locate the installed binary, detect its version (matching your existing SHA-256+version pinning approach), and know where "the original" lives for backup/rollback.

6. **PATH and multi-install conflicts are a known mess on this platform already** — WinGet, npm-global, and the native installer can all coexist and shadow each other (the docs' own troubleshooting section warns about "leftover shell alias from an older installer"). A patching tool should be defensive about *which* `claude` is actually on PATH before it touches anything, more so than on macOS where Homebrew/native rarely collide as messily.

7. **Sandboxing absence on native Windows is orthogonal but worth flagging to users**: if your tool's smoke-tests or install flow assume any OS-level sandbox parity with macOS, note that native Windows Claude Code has no Seatbelt/bubblewrap equivalent at all — WSL2 is the only sandboxed option on Windows today.
